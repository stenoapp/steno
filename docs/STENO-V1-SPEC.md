# Steno — v1.0 Specification

**Status:** Draft v0.1
**Prerequisite reading:** STENO-VISION.md
**Audience:** The coding agent building Steno v1.0

---

## 1. Scope of v1.0

A pan-OS desktop application that:

1. Records audio from a meeting (system audio + microphone, mixed)
2. Transcribes that audio locally using whisper.cpp or Parakeet
3. Performs best-effort speaker diarization
4. Generates a structured summary using a locally-running or BYOK LLM
5. Writes the transcript and summary as markdown files to a user-chosen folder
6. Indexes those files in SQLite for fast search
7. Exposes a local MCP server so agents on the same machine can query meetings, transcripts, and summaries
8. Ships as signed installers for macOS, Windows, and Linux

Out of scope for v1.0: screen recording, video capture, calendar integration, template editor UI (a default template ships; user-customisable templates come in v1.2), audio import (v1.3).

## 2. Architectural foundations

### 2.1 Stack

- **App shell:** Tauri 2.x
- **Backend language:** Rust
- **Frontend language:** TypeScript + React (or Svelte — agent chooses based on Tauri ecosystem maturity at build time; document the decision)
- **Persistence layer 1 — canonical:** Markdown files on the filesystem
- **Persistence layer 2 — index:** SQLite via `rusqlite`
- **Search:** SQLite FTS5 over indexed content
- **Audio capture:** `cpal` cross-platform crate, with OS-specific paths for system audio
- **Transcription:** `whisper.cpp` bindings (via `whisper-rs` or direct FFI); optional Parakeet integration where supported
- **LLM client:** Plain HTTP via `reqwest`. No SDK dependencies.
- **MCP server:** Implement to MCP spec; expose over local Unix socket / named pipe with simple HTTP fallback

### 2.2 The two-store model — non-negotiable

This is the single most important architectural decision. Implement it correctly and many other things become simple.

**Markdown files are the source of truth. SQLite is a derived index. Every byte in SQLite must be reconstructable from the markdown files.**

Concretely:

- If a user deletes the SQLite database, Steno rebuilds it on next launch by scanning the markdown folder
- If a user edits a markdown file in another editor (Obsidian, vim, VS Code), Steno detects the change on next launch (or via a file watcher when running) and re-indexes
- If a user moves a markdown file to another folder under Steno's root, the index updates the path
- The app never writes meeting content *only* to SQLite. SQLite holds: file paths, indexed text, embeddings (later), timestamps, and structural metadata derived from the markdown frontmatter.

The user can delete the app, keep the folder, install the app on a new machine, point it at the folder, and have everything back.

### 2.3 Folder layout

Default root: `~/Documents/Steno/` (configurable on first run).

```
Steno/
├── .steno/
│   ├── index.db              # SQLite index — derived, regeneratable
│   ├── audio/                # Raw audio recordings (compressed)
│   │   ├── 2026-05-18-1430-team-standup.opus
│   │   └── ...
│   ├── config.toml           # App config
│   └── models/               # Downloaded Whisper/Parakeet models
├── 2026/
│   └── 05/
│       ├── 2026-05-18-1430-team-standup.md
│       ├── 2026-05-18-1600-customer-acme.md
│       └── ...
└── templates/                # User templates (v1.2+); shipped defaults live in app bundle
```

The `Steno/` root is the folder the user picks. Everything outside `.steno/` is canonical and editable. Everything inside `.steno/` is derived or operational.

This layout works as an Obsidian vault out of the box. A user can open `~/Documents/Steno/` in Obsidian and immediately see, link, and tag their meetings.

### 2.4 Markdown file shape

Every meeting markdown file has YAML frontmatter and a structured body. The frontmatter is what SQLite indexes against; the body is what humans read.

```markdown
---
steno_version: 1
id: 01HXY...                          # ULID
created: 2026-05-18T14:30:00+05:30
duration_seconds: 2847
title: Team Standup
audio_file: .steno/audio/2026-05-18-1430-team-standup.opus
participants:
  - name: Chirag
    role: facilitator
  - name: Speaker 2
  - name: Speaker 3
model:
  transcription: whisper-large-v3
  summary: llama-3.1-70b@ollama
tags: []
---

# Team Standup

## Summary

Three-line synopsis generated by the LLM.

## Key Points

- Point one
- Point two

## Action Items

- [ ] @Chirag to ship the spec by Friday
- [ ] @Speaker2 to review the PR

## Decisions

- Decided to defer the auth refactor to Q3

## Transcript

[14:30:02] **Chirag:** Right, let's get started...
[14:30:15] **Speaker 2:** I wanted to flag...
```

The transcript section is the longest part of the file but appears last so the summary is immediately visible when opened in any markdown viewer.

### 2.5 SQLite schema (sketch — agent finalises)

```sql
CREATE TABLE meetings (
    id TEXT PRIMARY KEY,            -- ULID from frontmatter
    file_path TEXT NOT NULL,        -- relative to Steno root
    title TEXT NOT NULL,
    created_at TEXT NOT NULL,
    duration_seconds INTEGER,
    last_indexed_at TEXT NOT NULL,
    file_hash TEXT NOT NULL         -- for change detection
);

CREATE VIRTUAL TABLE meeting_fts USING fts5(
    id UNINDEXED,
    title,
    summary,
    transcript,
    content='meetings_content'
);

CREATE TABLE participants (
    meeting_id TEXT REFERENCES meetings(id),
    name TEXT,
    role TEXT
);

CREATE TABLE action_items (
    meeting_id TEXT REFERENCES meetings(id),
    text TEXT,
    assignee TEXT,
    done INTEGER DEFAULT 0
);
```

Index rebuild is a documented user-facing operation, not a hidden recovery path.

## 3. Audio capture

### 3.1 What gets captured

Two streams, mixed at record-time:

1. **System audio output** — what the user hears (the other participants)
2. **Microphone input** — what the user says

Mixing strategy: capture both as separate channels, mix in real time to a single mono stream for transcription, optionally save the dual-channel original for re-processing later (v1.3 territory).

### 3.2 Per-OS capture mechanism

| OS | System audio | Microphone | Notes |
|---|---|---|---|
| macOS | `ScreenCaptureKit` (macOS 13+) | `cpal` / AVAudioEngine | Requires Screen Recording permission; user prompted on first use |
| Windows | WASAPI loopback | WASAPI | No special permissions needed beyond mic |
| Linux | PipeWire (preferred) / PulseAudio fallback | cpal | Some distros need explicit module loading for monitor sources; document in setup |

If a platform's system audio capture fails or is unavailable, Steno records mic-only and shows a visible warning. It does not silently degrade.

### 3.3 Storage format

Opus in an Ogg container. Quality target: 24 kbps mono is sufficient for transcription; 32 kbps gives audible-quality playback for users who want it. Configurable; default 32 kbps.

A 1-hour meeting at 32 kbps is ~14 MB. A user with 200 meetings/year accumulates ~3 GB/year. Acceptable.

## 4. Transcription

### 4.1 Models

Ship with no models bundled. On first run, prompt the user to download a Whisper model. Offer:

- `tiny` (75 MB, fast, lower quality) — for older machines
- `base` (150 MB) — default recommendation
- `small` (500 MB) — better quality
- `medium` (1.5 GB) — recommended for serious use
- `large-v3` (3 GB) — best quality, slower

Download from Hugging Face mirrors. Show progress. Verify SHA256.

Parakeet support is a v1.0 stretch goal — include if shippable, defer to v1.1 if not.

### 4.2 Live vs post-hoc

v1.0 ships post-hoc transcription only: record → stop → transcribe → summarise. Live transcription is technically possible with whisper.cpp's streaming mode but adds complexity (chunking, partial-hypothesis UI, performance tuning). Push to v1.1.

This decision is reversible. Architect the pipeline so live mode can be added without restructuring.

### 4.3 Diarization

Use a lightweight diarization model (e.g., `sherpa-onnx` speaker segmentation) post-transcription. Output is "Speaker 1, Speaker 2, ..." labels. User can rename speakers in the meeting view; renames are written back to the markdown file.

Diarization is best-effort and flagged as such in the UI. Quality varies dramatically with meeting audio quality.

## 5. LLM integration

### 5.1 First-class providers

Three providers get first-class UX (auto-detect, status indicator, no manual URL entry):

1. **Ollama** — detect at `http://localhost:11434`; list available models; recommend models suitable for summarisation
2. **LM Studio** — detect at `http://localhost:1234`; same shape
3. **Generic OpenAI-compatible endpoint** — user enters URL + optional API key; everything else fits this shape (OpenRouter, Anthropic via proxy, OpenAI directly, llamafile, etc.)

The first-class detection is a wedge against Meetily, which treats all three as the same generic "endpoint config."

### 5.2 Prompts

Ship a default summarisation prompt that produces the markdown structure shown in §2.4. Prompt lives in code as a constant in v1.0; becomes user-editable templates in v1.2.

The prompt asks for: a 3-line summary, key points (bulleted), action items (with assignees when identifiable from transcript), decisions, and follow-ups. The prompt explicitly forbids inventing content not present in the transcript.

### 5.3 Failure modes

LLM call fails → meeting still saves with transcript and a marker that summarisation failed. User can retry summarisation from the meeting view. Transcript is never lost because of an LLM failure.

LLM produces malformed output → fall back to a "raw response" mode that saves whatever the LLM returned verbatim under a "## Notes" heading, with a warning.

### 5.4 Network posture

The only network calls Steno makes are to user-configured LLM endpoints, and to Hugging Face mirrors for model downloads (with the URL shown to the user before download). No analytics, no update checks (v1.0 uses manual updates; auto-update considered for v1.1), no crash reporting.

## 6. MCP server

Steno exposes a Model Context Protocol server on a local Unix socket (macOS/Linux) or named pipe (Windows). Other agents on the user's machine can connect and call:

**Tools:**
- `list_meetings(since, until, limit)` — return meeting metadata
- `get_meeting(id)` — return full markdown content of a meeting
- `search_meetings(query, limit)` — FTS5 search across summaries and transcripts
- `get_action_items(assignee, done, since)` — structured query over extracted action items

**Resources:**
- Each meeting exposed as `steno://meeting/{id}` returning the markdown file

The MCP server runs whenever the Steno app is running. It does not run as a separate background daemon in v1.0 (that's v1.1 — running headless).

Authentication: local socket access only in v1.0. No HTTP exposure. No remote access.

## 7. UI / UX

### 7.1 Surfaces

1. **Main window** — list of past meetings (newest first), search bar, "Start Recording" button
2. **Recording view** — large recording indicator, elapsed time, audio level meters for system + mic, stop button, "I need to pause" toggle
3. **Meeting view** — rendered markdown of the meeting; "Edit in external editor" opens the .md file in the user's default editor; "Re-summarise" / "Re-transcribe" actions
4. **Settings** — folder location, audio devices, LLM provider config, Whisper model management
5. **System tray / menu bar icon** — quick access to start/stop recording, status indicator (idle / recording / processing)

### 7.2 Keyboard-first

Every action has a keyboard shortcut. Power users should be able to start a recording, stop it, and open the resulting meeting without touching a mouse. Document shortcuts in a visible help panel.

### 7.3 What we explicitly do not build in v1.0

- Onboarding wizard beyond the first-run model download
- Tutorial videos, tooltips, or animated walkthroughs
- "Welcome" emails, in-app announcements, feature discovery prompts
- Telemetry-driven UX improvements (no telemetry)
- A web dashboard (there is no web)
- Account creation, login, password reset (there are no accounts)

## 8. Settings persistence

`~/.config/steno/config.toml` on Linux, `~/Library/Application Support/Steno/config.toml` on macOS, `%APPDATA%\Steno\config.toml` on Windows. Plus a duplicate `.steno/config.toml` inside the meetings folder for folder-portable settings (which LLM provider, which prompt template). App config in the OS location is for UI state; folder config is for content-shaping decisions that should travel with the folder.

This split matters: if a user puts their Steno folder in iCloud Drive and opens it from a second machine, the content-shaping settings come with it.

---

## 9. Error model and observability

No crash reporter. No telemetry. Errors surface to the user via:

1. **In-app error banner** for user-actionable problems (mic disconnected, LLM unreachable)
2. **Local log file** at `.steno/logs/steno.log` — rotated, capped at 50 MB total
3. **"Copy diagnostic info" button** in Settings that copies recent log entries + system info to the clipboard for pasting into a GitHub issue

The log file never contains transcript content or meeting audio. It contains errors, warnings, and operational metadata only. This rule is checked in code review.

## 10. Performance targets (informational, not contractual)

For a 1-hour meeting on a 2023-era laptop with `whisper-base`:

- Transcription completes within 10 minutes of recording end
- Summarisation completes within 2 minutes of transcription end (depends on LLM)
- App memory footprint while idle: under 200 MB
- App memory footprint while recording: under 400 MB
- Cold-start time: under 3 seconds on SSD-equipped machines

Targets are not gates. Ship if functionality is right and performance is in a reasonable neighbourhood; tune in v1.x.

---

## 11. Security and privacy posture

- No data leaves the device except to user-configured LLM endpoints and Hugging Face for model downloads
- Audio files are not encrypted at rest in v1.0 (filesystem-level encryption is the user's responsibility); revisit in v1.x
- The user is told, in the Settings page, exactly which network endpoints Steno can talk to, and the active configured endpoints are shown
- Network requests to LLM endpoints respect system proxy settings
- The app does not auto-update in v1.0

## 12. Dependency licence policy — hard rule

Every dependency added to Steno must be under: MIT, Apache-2.0, BSD (2 or 3 clause), ISC, MPL-2.0, or public domain / Unlicense / CC0.

**Forbidden licences:** GPL (any version), AGPL, LGPL (because of the static-linking complications with Rust), SSPL, or any custom non-standard licence.

This rule exists to preserve future paid SKU optionality. The agent must verify the licence of every dependency before adding it, and must document the licence in a `LICENSES.md` file at the repo root.

If a desirable feature requires a GPL-licensed component, the agent stops and asks. Do not work around this rule silently.

## 14. References — repos to study, not copy

The agent reads these for technique. None of their code is imported.

- `Zackriya-Solutions/meetily` — closest reference; study audio capture and Whisper integration in their Rust backend
- `fastrepl/anarlog` (and the original `hyprnote`) — study LLM provider abstraction and UI patterns
- `mediar-ai/screenpipe` — study the continuous-capture architecture for v2 inspiration only; do not import patterns into v1.0
- `huggingface/candle` and `tazz4843/whisper-rs` — Whisper bindings for Rust
- `RustAudio/cpal` — cross-platform audio I/O reference and API
- `k2-fsa/sherpa-onnx` — diarization options
- Anthropic's MCP documentation and reference servers — for protocol implementation

When studying a reference repo, the agent writes notes in `docs/REFERENCES.md` describing what was learned and what was decided. This is the audit trail that proves we built fresh.
